Waterfall
Click any of the following to find out more about each stage:
  • 1. Requirements
    The requirements are looked over. The client will explain excatly what they want, and the development team will take notes in preperation for analysis. If this stage is not done well, the whole project can be a waste of time because there isn't ever much communication with the client
  • 2. Analysis
    During analysis, the project's feesability is questioned. More specifically, the legal, finantial and timeframes are looked over, and the price is set. Beyond this point, the project is likely to cost much more, and so it must be done to the fullest extent.
  • 3. Design
    Design consists of putting the project together in the form of tree diagrams flow charts etc. The logic behind each planned function is decided, and safety aspects of software are gone over to ensure the final product will be higher quality.
  • 4. Coding
    The coding process is simply creating the application. Code by each developer is submitted and pushed to a centeral hub, where testing may occur however it is usually saved for the testing process.
  • 5. Testing
    During the testing phase, white (line by line) box or black (console/function based only) testing. This is usually a very extensive phase of the waterfall process along with documentation, and is considered to give code produced by this development method it high quality.
  • 6. maintenance
    Maintaining the code simply means updating it to fix errors or create slight adaptions. The maintenance phase can often encompass other parts of the cycle within it.

Agile Methods
Agile software development describes a set of principles for software development under which requirements and solutions evolve through the collaborative effort of self-organizing cross-functional teams. It advocates adaptive planning, evolutionary development, early delivery, and continuous improvement, and it encourages rapid and flexible response to change.
Rapid Application Development
RAD approaches to software development put less emphasis on planning and more on process. In contrast to the waterfall model, RAD approaches emphasize adaptability and the necessity of adjusting requirements in response to knowledge gained as the project progresses. Prototypes are often used in addition to or sometimes even in place of design specifications.

Extreme programming (XP) is a software development methodology which is intended to improve software quality and responsiveness to changing customer requirements. As a type of agile software development, it advocates frequent "releases" in short development cycles, which is intended to improve productivity and introduce checkpoints at which new customer requirements can be adopted. Other elements of extreme programming include: programming in pairs or doing extensive code review, unit testing of all code, avoiding programming of features until they are actually needed, a flat management structure, code simplicity and clarity, expecting changes in the customer's requirements as time passes and the problem is better understood, and frequent communication with the customer and among programmers. The methodology takes its name from the idea that the beneficial elements of traditional software engineering practices are taken to "extreme" levels. As an example, code reviews are considered a beneficial practice; taken to the extreme, code can be reviewed continuously, i.e. the practice of pair programming.
Scrum is an iterative and incremental agile software development framework for managing product development. It defines "a flexible, holistic product development strategy where a development team works as a unit to reach a common goal", challenges assumptions of the "traditional, sequential approach" to product development, and enables teams to self-organize by encouraging physical co-location or close online collaboration of all team members, as well as daily face-to-face communication among all team members and disciplines involved. A key principle of Scrum is its recognition that during product development, the customers can change their minds about what they want and need (often called requirements volatility), and that unpredicted challenges cannot be easily addressed in a traditional predictive or planned manner. Disciplined agile delivery (DAD) is a process decision framework that enables simplified process decisions around incremental and iterative solution delivery. DAD builds on the many practices espoused by advocates of agile software development, including Scrum, agile modeling, lean software development, and others. DAD Follows a iterative model, however it is prepended with Inception, Construction and Transition. Each of these phases are iterative within themselves. Following that there is an OnGoing phase in which the product is adaptively developed and maintained.
